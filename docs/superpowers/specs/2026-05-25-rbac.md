# Spec: RBAC (TODO #1)

Status: Draft · Date: 2026-05-25 · Sub-project C of the TODO decomposition

## Problem

The whole `/v1` API is gated by one shared bearer (`DENIA_ADMIN_TOKEN`,
`require_admin_token` in `src/app.rs`). There are no users, no roles, and no way
to give someone scoped, read-only, or per-project access. Multi-operator use is
impossible.

## Goal

Introduce users with password login, bearer tokens (expiring session tokens for
the console + long-lived API tokens for CI), and **project-scoped roles**
(admin / operator / viewer) plus a global super-admin. Authorize every `/v1`
route by the caller's role on the relevant project.

## Decisions

- **Auth model:** users have an argon2 password hash. `POST /v1/auth/login`
  issues an opaque session token (random, stored as a SHA-256 hash, with
  expiry). Separate named **API tokens** (long-lived) for automation. Both map to
  a user. `DENIA_ADMIN_TOKEN` remains as a bootstrap **super-admin** bearer so the
  node is usable before any user exists.
- **Roles are project-scoped:** a `(user, project, role)` membership.
  `viewer` = read project resources; `operator` = deploy/manage services in the
  project; `admin` = manage members + everything in the project. A user-level
  `is_super_admin` flag grants node-wide access (manage users, all projects).
- **Authorization:** middleware resolves the bearer -> user. Global routes
  (`/v1/users`) require super-admin. Project-scoped routes resolve the target
  project (directly, or via the service's `project_id` from sub-project B) and
  require the caller's membership role >= the route's required role.
- **Hashing:** passwords via `argon2`; tokens are high-entropy random, stored as
  SHA-256 (no argon2 needed for tokens). New deps: `argon2`, `rand`,
  `sha2` (or reuse an existing hash).

## Dependencies

Requires sub-project B (Projects + `service.project_id`) for project resolution
and memberships. Build B first.

## Data model (`src/domain.rs`)

- `User { id, username (unique), password_hash, is_super_admin, created_at }`.
- `Role` enum: `Admin | Operator | Viewer` (ordered for >= comparison).
  Serialize roles as snake_case strings (`admin`, `operator`, `viewer`) so the
  frontend does not depend on Rust enum casing.
- `ProjectMembership { user_id, project_id, role }`.
- `ApiToken { id, user_id, name, token_hash, created_at, last_used_at }`.
- `Session { token_hash, user_id, expires_at }`.
- Never serialize `password_hash`/`token_hash` outward.
- Public API shape for identity:
  `Me { principal, is_super_admin, memberships }`, where `principal` is
  `{ kind: "user", user }` for real users or `{ kind: "bootstrap" }` for the
  `DENIA_ADMIN_TOKEN` synthetic principal.

## State (`src/state.rs`, versioned migration from B)

- Tables: `users`, `sessions`, `api_tokens`, `project_members`.
- Methods: user CRUD; `verify_login(username, password) -> User`; session
  create/lookup/delete + expiry purge; API token create/list/revoke + lookup;
  membership upsert/list/remove; `role_for(user, project) -> Option<Role>`.
- Seed: ensure a super-admin principal exists for the bootstrap env bearer.

## Auth + authorization (`src/auth.rs` new, `src/app.rs`)

- `Principal { user_id, is_super_admin }` resolved from the bearer (env bootstrap
  token -> synthetic super-admin; session/API token -> user).
- Replace `require_admin_token` with `require_auth` producing a `Principal`
  extension; add `require_super_admin` and `require_project_role(min: Role)`.
- Project-scoped handlers resolve the project (path `project_id`, or the
  service's project) and check `role_for(principal, project) >= min` (super-admin
  bypasses).

## API (`/v1`, bearer-protected)

- `POST /v1/auth/login` {username, password} -> {token, expires_at}.
- `POST /v1/auth/logout` (revoke current session).
- `GET /v1/me` -> `Me` (`principal`, `is_super_admin`, `memberships`).
- `GET/POST /v1/users`, `DELETE /v1/users/{id}` (super-admin).
- `GET/POST /v1/api-tokens`, `DELETE /v1/api-tokens/{id}` (current user).
- `GET/POST /v1/projects/{id}/members`, `DELETE /v1/projects/{id}/members/{user_id}`
  (project admin or super-admin).
- Existing resource routes gain a required role (read=viewer, mutate=operator,
  member/project admin=admin).

## Errors / edge cases

- Bad login -> 401, constant-time-ish failure (argon2 verify always runs).
- Expired/unknown token -> 401. Insufficient role -> 403.
- Last super-admin cannot be deleted/demoted -> 409.
- Bootstrap env bearer keeps working even with zero users.
- Bootstrap env bearer can manage users/projects but cannot mint self API tokens
  because it has no persisted user id; create a real super-admin for that.
- Token shown once on creation; only the hash is stored.

## Success criteria

- Login issues a working session token; logout revokes it.
- A viewer can read but not deploy; an operator can deploy in their project but
  not in others; a project admin manages members; super-admin manages users.
- The env bootstrap bearer still works.

## Testing

- Domain: `Role` ordering; redaction of secrets in serde.
- State: login verify (good/bad), session expiry, token lookup, `role_for`,
  last-super-admin guard, migration.
- Auth: principal resolution (env/session/api); `require_project_role` allow/deny;
  super-admin bypass.
- API: login/logout/me; users CRUD gated; members CRUD; a viewer token blocked
  from a deploy (403); cross-project denial.

## Out of scope

OAuth/SSO, MFA, fine-grained per-resource ACLs, audit log (later), rate limiting.
Frontend covered by the companion spec.
