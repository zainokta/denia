# Spec: Jobs / Task Scheduler (TODO #6)

Status: Draft · Date: 2026-05-25 · Sub-project F of the TODO decomposition

## Problem

Denia only runs long-lived services. There is no way to run a workload to
completion (a one-off task) or on a schedule, the way Kubernetes `Job`/`CronJob`
or Cloud Run Jobs/Tasks do. TODO #6 asks for run-to-completion jobs and scheduled
execution.

## Goal

A Job is a workload that runs to completion (not kept alive, no ingress), run
either manually or on a cron schedule. Each execution is a JobRun with a status
and exit code. Built on the existing Linux runtime isolation; single-node.

## Decisions

- **Execution reuses the runtime:** extend the runtime with a
  **run-to-completion** path that starts the workload with the same
  userns/cgroup isolation as services, waits for it to exit, and returns the exit
  code. No socket, no bridge port, no Traefik route, no health check.
- **Scheduling is in-process:** a tokio scheduler task evaluates each job's cron
  schedule and triggers JobRuns. Denia accepts standard 5-field cron
  (`*/5 * * * *`) and normalizes it for the parser if the chosen crate requires
  seconds. Schedules are evaluated in UTC. New dep: a cron parser (`cron` crate
  or equivalent). No systemd/crontab.
- **Concurrency policy:** `Forbid` by default — if the previous run of a job is
  still active when the next fire time arrives, skip it (record a `Skipped` run).
- **Retries:** `max_retries` (default 0); a failed run retries up to the limit
  with a fixed backoff before being marked `Failed`.
- **Scheduler cursor is persisted:** scheduled jobs track `next_run_at` and
  `last_enqueued_at` so a restart or fast scheduler tick cannot enqueue the same
  cron fire twice.
- **Project-scoped:** a Job belongs to a project (sub-project B); RBAC (C) gates
  it later (operator to run, viewer to read).
- **No raw secrets in jobs:** `env` is for non-secret key/value overrides only.
  Secret material must remain referenced through Denia's secret/SOPS model; do
  not store raw credentials in the job row.

## Dependencies

Sub-project B (Projects, `project_id`, versioned migrations). RBAC (C) gates the
endpoints when present.

## Data model (`src/domain.rs`)

- `Job { id, project_id, name (unique per project), source: ServiceSource,
  command: Option<Vec<String>>, env: Vec<(String,String)>,
  resource_limits: ResourceLimits, schedule: Option<String> /* cron */,
  next_run_at: Option<DateTime>, last_enqueued_at: Option<DateTime>,
  max_retries: u32, created_at }`.
- `JobRunStatus`: `Pending | Running | Succeeded | Failed | Skipped`.
- `JobRun { id, job_id, status, attempt: u32, exit_code: Option<i32>,
  started_at: Option<...>, finished_at: Option<...>, created_at }`.
- Validate: non-empty name; if `schedule` set, it must parse as cron.
- Validate env keys with the same runtime rules as process manifests; reject
  invalid keys before persisting the job.
- `attempt` is the final attempt count for that run: initial attempt is `1`,
  each retry increments it, and `Skipped` runs use `0`.

## Runtime (`src/runtime.rs`)

- Add a run-to-completion entry: `run_to_completion(req) -> JobOutcome { exit_code }`.
  Split common runtime planning from the service-specific socket-proxy path:
  do not call the current service `plan` unchanged because it injects
  `DENIA_SERVICE_SOCKET` and the socket proxy wrapper. The job plan uses rootfs +
  userns + cgroup, no socket/bridge, spawns the child, `await`s its exit, captures
  the status, then cleans up the cgroup/deployment dir.
- A `JobRunRequest` (job id, run id, artifact, command, env, limits) mirrors
  `RuntimeStartRequest` minus ingress fields.
- Runtime merges the image `process.json` with job overrides at execution time:
  `command` replaces argv when set, and job env overrides image env by key. Do
  not mutate shared `process.json` in the rootfs bundle per run.

## Scheduler (`src/scheduler.rs`, new)

- Spawned from `main`/`AppState`; holds the `SqliteStore`, runtime,
  `ArtifactAcquirer`, and `CommandRunner` so scheduled runs can materialize the
  rootfs bundle just like deploys.
- Loop: wake on the nearest persisted `next_run_at`; atomically advance
  `last_enqueued_at`/`next_run_at` before spawning work. For each due job, if an
  active run exists (Forbid), create a `Skipped` JobRun for that fire time.
  Otherwise create a `Pending` JobRun and spawn background execution, then mark
  `Running`, recording the outcome (+ retries with backoff). Manual triggers use
  the same enqueue/spawn path and return immediately.
- Graceful shutdown must be added in `main` with `axum::serve(...).with_graceful_shutdown(...)`
  and a shared cancellation signal for the scheduler; there is no existing
  shutdown hook today.

## State (`src/state.rs`, versioned migration)

- Tables `jobs`, `job_runs`.
- `put_job`/`get_job`/`list_jobs(project)`/`delete_job`; `create_run`,
  `update_run(status, exit_code, finished_at)`, `list_runs(job)`,
  `active_run(job) -> Option` (Pending or Running), `claim_due_jobs(now)`,
  `advance_job_schedule(job, fire_time)`, and `fail_orphan_runs()`.

## API (`/v1`, bearer-protected; role-gated once RBAC lands)

- `GET /v1/jobs?project_id=...` -> job summaries newest/name sorted, including
  `latest_run: Option<JobRun>` and `next_run_at`.
- `POST /v1/jobs`, `GET/DELETE /v1/jobs/{id}`.
- `POST /v1/jobs/{id}/run` -> enqueues a JobRun immediately (202 + run id) and
  does not wait for job completion.
- `GET /v1/jobs/{id}/runs` -> run history (newest first).

## Errors / edge cases

- Invalid cron -> 400 on create.
- Invalid env key or raw secret-looking field -> 400 on create/update.
- Manual run while a run is active (Forbid) -> 409.
- Job with no source/artifact -> validation error.
- Run failure path records `Failed` with the exit code after retries exhausted.
- Scheduler restart: in-flight runs at crash are reconciled to `Failed` on
  startup (no orphan `Running` rows).
- Delete while a run is active -> 409; cancellation is out of scope.

## Success criteria

- A manual run executes the workload to completion and records exit code +
  status.
- A scheduled job fires on its cron, respects Forbid concurrency, and retries up
  to `max_retries`.
- Run history is queryable per job.

## Testing

- Domain: cron validation; status transitions.
- Runtime: `run_to_completion` returns the child exit code (gated privileged
  test for the real namespace path; a `FakeRuntime` variant for unit logic).
- Scheduler: due-time selection; persisted cursor prevents duplicate enqueue;
  Forbid skip; retry/backoff; manual trigger returns before completion (with a
  fake clock + fake runtime/acquirer).
- State: jobs/runs CRUD; `active_run`; migration.
- API: create/list/delete; manual run; run history; invalid cron 400.

## Out of scope

Job dependencies/DAGs, parallelism/array jobs, timeouts/deadlines (could be a
fast-follow), log retention policy. Frontend covered by the companion spec.
